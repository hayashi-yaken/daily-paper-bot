## 概要

OpenReview から取得した論文の Abstract を Azure AI Translator (v3.0 `/translate`) で日本語に機械翻訳し、投稿の主役を訳に置きつつ原文も別表現で確認できるようにする。Slack では原文をスレッド子に出し、Discord では spoiler で同梱する。翻訳機能は `TRANSLATE_ENABLED` フラグで ON/OFF を切り替え、Translator API が失敗した場合は原文だけで投稿を続行する（WARN ログのみ）。

あわせて、ヘッダリンクが常にカンファレンス TOP を指す現行仕様を改修し、その日の論文ページ (`forum?id={paperID}`) を指すようにする。本文末尾の `*Link*:` は `*PDF*:` に改名し、PDF が取得できているときだけ出力するように整理する。

## 方針

- 翻訳は「整形」や「送信」と独立した関心事として `internal/translator` パッケージに切り出し、`formatter` / `notifier` には機械翻訳の存在を持ち込まない（main.go で結果を受け渡す）。
- 翻訳機能は ON/OFF できる任意機能とする。デフォルト OFF。`TRANSLATE_ENABLED=true` を明示したときだけ Key/Region を必須にする。
- Translator API の失敗は投稿全体の失敗にしない。WARN ログを残しつつ原文だけで投稿を続行することで、毎日動くバッチとしての可用性を優先する。
- リトライ・キャッシュは導入しない。バッチの 1 回呼び出しで毎回直接 Azure を叩くシンプルな実装にする。
- Slack でのスレッド子投稿失敗も親投稿が成功している以上「失敗」とは扱わず、WARN ログで吸収する。
- ヘッダリンクの差し替え・末尾 Link → PDF の改修は `formatter.go` を触る今回のスコープに含める。翻訳機能の有無と独立に常時適用される。
- 標準ライブラリの `net/http` のみで Azure AI Translator を呼び、追加の SDK 依存を導入しない。

## 事前準備（ユーザー作業）

1. Azure ポータルで「Translator」リソースを作成する（無料枠 F0 で 200 万文字/月）
2. リソースのリージョンと Key 1 を控える
3. GitHub リポジトリの **Settings > Secrets and variables > Actions** で以下を登録する
   - `AZURE_TRANSLATOR_KEY`: 取得したサブスクリプションキー
   - `AZURE_TRANSLATOR_REGION`: リソースのリージョン（例: `japaneast`）
4. `TRANSLATE_ENABLED=true` を `.env` または GitHub Secrets に追加する（既存ボットの環境変数管理に揃える）

## 備考

- 翻訳結果は永続化せず、毎日実行のたびに Azure を直接叩く。F0 枠（200 万文字/月）で十分余裕がある想定。
- ヘッダリンクと PDF 行の改修は翻訳機能とは独立した UX 改善だが、`formatter.go` を触るスコープに同梱して 1 つのチケットで扱う。
